# 05. PRD 작성 보고서 - Prompting

## 1. 사용 프롬프트

### 1.1 전체 PRD 작성 요청

```text
당신은 제품 요구사항 문서(PRD) 전문가입니다.
워크스페이스: @c:\dev\UnitConverter_A02\ 
⚠️ 코드·클래스 설계·빌드 스크립트는 작성하지 마십시오. 문서만.
전제: Phase 4에서 완성된 Epic/User Stories/Gherkin을 기반으로 작성하라.

# 1. 프로젝트 개요
  1.1 한 줄 목적문 (What / Who / Why)
  1.2 배경 및 문제 정의 (관찰 관점)
  1.3 목표 3~5개 (측정 가능하게)
  1.4 비목표(Non-Goal) 2~3개

# 2. 사용자 및 이해관계자
  2.1 타깃 사용자 (페르소나 1개)
  2.2 주요 사용 시나리오 2~3개 (Phase 4 Journey 기반)

# 3. 기능 요구사항
  3.1 핵심 기능 목록 (우선순위: 필수/권장/선택 구분)
  3.2 기능별 입·출력 계약 (문자열 계약 수준, 코드 X)
  3.3 제약 사항 (입력 형식, 허용 단위, 음수 정책, 미지원 단위 처리)
  → Phase 4 Gherkin의 Background Given과 반드시 일치시킬 것

# 4. 비기능 요구사항
  4.1 기술 스택: Java 17, Gradle/Maven, JUnit 5
  4.2 아키텍처 원칙 (OCP/SRP, BCE 레이어, 의존성 방향)
  4.3 테스트 커버리지 목표 (Domain/Boundary/Data 수치)
  4.4 확장성 원칙 (새 단위 추가 시 기존 코드 변경 최소화)

# 5. 데이터 요구사항
  5.1 단위 비율 상수 (meter 허브 기준)
  5.2 설정 외부화 방식 (JSON/YAML 또는 메모리 기본값)
  5.3 동적 단위 등록 계약 (register: 형식 명시)

# 6. 출력 요구사항
  6.1 콘솔 기본 포맷 (필드·순서·소수 자릿수 정책)
  6.2 JSON 출력 스키마 (확장 시)
  6.3 CSV 출력 스키마 (확장 시)
  6.4 표(Table) 출력 스키마 (확장 시)

# 7. 성공 지표
  7.1 인수 기준 3~5개 (체크박스, 테스트 가능하게)
       → Phase 4 Story의 Acceptance Criteria와 일치 여부 확인
  7.2 회귀 보호 규칙 (계약 변경 금지 정책)

# 8. 용어 정의 (Glossary)
  주요 도메인 용어 5개 이상

출력: 구조화된 Markdown만. 모호어("적절히") 금지.
모든 계약·기준은 테스트 가능한 문장으로 작성.
```

### 1.2 PRD 섹션 1~2 작성 요청

```text
Phase 4 Epic을 기반으로 PRD의 1~2번 섹션(개요, 사용자)만 작성하라.
목표는 측정 가능해야 하고, 비목표는 경계를 명확히 하라.
구현 금지.
```

### 1.3 PRD 섹션 3 작성 요청

```text
Phase 4 User Stories를 기반으로 기능 요구사항(섹션 3)을 작성하라.
입출력 계약은 문자열 수준으로만 정의하고, 코드는 작성하지 말 것.
우선순위(필수/권장/선택)를 반드시 구분하라.
```

### 1.4 PRD와 Phase 4 비교 요청

```text
완성된 PRD와 Phase 4(Epic/Story/Gherkin/체크리스트)를 비교하라.
보고만 (수정 금지):
1) Phase 4 Story에 있으나 PRD에 빠진 기능
2) PRD에 있으나 Phase 4 Gherkin과 계약이 불일치하는 항목
3) PRD → Phase 6(README) 이관 시 누락 위험 항목
bullet만. 코드 금지.
```

### 1.5 PRD 파일 내보내기 확인 요청

```text
지금까지 진행한 내용을 docs/PRD.md 파일로 내보내줘
```

### 1.6 보고서와 Prompting 내보내기 요청

```text
이번세션에서 진행한 내용을 report 폴더를 만들어서 05_PRD 작성 
_보고서 파일로 내보내주고, 
Prompting 폴더를 만든 후 프롬프트를 포함해서 05_PRD 작성_보고서-Prompting.md 파일로 내보내줘
```

파일명은 기존 보고서 명명 규칙과 Markdown 확장자를 기준으로 다음과 같이 해석했다.

| 요청 표현 | 생성 파일 |
|---|---|
| `05_PRD 작성 _보고서 파일` | `report/05_PRD 작성_보고서.md` |
| `05_PRD 작성_보고서-Prompting.md` | `Prompting/05_PRD 작성_보고서-Prompting.md` |

## 2. 프롬프트 해석 기준

이번 세션의 요청은 다음 기준으로 해석했다.

| 기준 | 해석 |
|---|---|
| Phase 4 기반 | `docs/requirements.md`의 Epic, Journey, Story, Gherkin, 체크리스트를 기준으로 삼는다. |
| 구현 금지 | Java 코드, 클래스 설계, 메서드 시그니처, 패키지 구조, 빌드 스크립트를 작성하지 않는다. |
| 문자열 계약 수준 | 기능별 입출력은 예시 문자열, 필드명 문자열, 오류 분류 문자열로만 표현한다. |
| 측정 가능성 | 목표, 인수 기준, 커버리지, 회귀 보호 규칙은 테스트 가능한 문장으로 작성한다. |
| 우선순위 구분 | 기능 요구사항은 필수, 권장, 선택으로 구분한다. |
| 수정 금지 요청 | 비교 요청에서는 파일을 수정하지 않고 결과만 보고한다. |

## 3. 단계별 산출물 요약

### 3.1 전체 PRD

PRD는 1~8번 섹션으로 구성했다.

| 섹션 | 산출물 |
|---:|---|
| 1 | 프로젝트 개요 |
| 2 | 사용자 및 이해관계자 |
| 3 | 기능 요구사항 |
| 4 | 비기능 요구사항 |
| 5 | 데이터 요구사항 |
| 6 | 출력 요구사항 |
| 7 | 성공 지표 |
| 8 | 용어 정의 |

### 3.2 개요와 사용자

Phase 4 Epic과 Journey를 기준으로 다음 내용을 반영했다.

| 항목 | 반영 기준 |
|---|---|
| 목적문 | What / Who / Why |
| 배경 | 관찰 관점 문제 정의 |
| 목표 | 측정 가능한 5개 목표 |
| 비목표 | 구현 산출물, 범위 밖 기능, 단위 카탈로그 확장 제외 |
| 사용자 | Java/클린 아키텍처 학습자 |
| 시나리오 | 문제 인식, 계약 정의, 도메인 분리, 검증, 회귀 보호 |

### 3.3 기능 요구사항

Phase 4 User Stories를 기준으로 섹션 3을 재작성했다.

| Story | PRD 반영 |
|---|---|
| S1 | 입력 검증, 숫자 값 검증, 음수 거부 |
| S2 | 기본 등록 단위, 등록 단위 기준 판단 |
| S3 | README 비율, meter 허브 교차 변환 |
| S4 | 콘솔, JSON, CSV, Table 출력 |
| S5 | 설정 로드 실패 분류 |
| S6 | 동적 단위 등록, 중복 단위명 거부 |

### 3.4 비교 보고

완성된 PRD와 Phase 4 산출물을 비교해 다음 위험을 보고했다.

| 구분 | 보고 내용 |
|---|---|
| Story 누락 | 뚜렷한 누락 없음 |
| Gherkin 불일치 | 오차 수치, 0 값 정책, 콘솔 반올림, 포맷별 실패 계약, 동적 등록 중복 정책, 설정 실패 세부 조건 |
| README 이관 위험 | 절대 오차, 0 값 정책, Table 스키마, 중복 단위명 거부, 설정 실패 구분, FR 번호 참조 |
