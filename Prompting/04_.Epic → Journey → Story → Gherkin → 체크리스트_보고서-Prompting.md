  Epic→Journey→Story→Gherkin 정합성 체크박스 표

## 2. 프롬프트 해석 기준

이번 세션의 요청은 다음 기준으로 해석했다.

| 기준 | 해석 |
|---|---|
| 구현·코드 금지 | Java 코드, 클래스 설계, 메서드 시그니처, 패키지 구조를 작성하지 않는다. |
| 저장소 범위 제한 | README의 기본 요구사항, 품질 요구사항, 추가 요구사항 안에서만 작성한다. |
| README 비율 고정 | `1 meter = 3.28084 feet`, `1 meter = 1.09361 yard`를 기준 계약으로 둔다. |
| 학습 목표 | 알고리즘 구현보다 계약 정의, 테스트 설계, 레이어 분리 학습을 우선한다. |
| 추적성 | Epic → Journey → Story → Gherkin → 체크리스트 흐름이 서로 대조 가능해야 한다. |

## 3. 단계별 산출물 요약

### 3.1 Epic

Epic 제목은 `확장 가능한 Java 단위 변환 학습 시스템`으로 고정했다.

성공 기준은 다음 측정 항목을 포함하도록 구성했다.

| 측정 항목 | 내용 |
|---|---|
| 커버리지 | 핵심 도메인 변환 로직과 입력 검증 분기 목표 |
| 계약 테스트 | README 비율, 입력 오류, unknown unit, 설정 오류 |
| 회귀 정책 | 회귀 결함 재현 테스트 선정의 |
| 확장성 | 새 단위나 출력 포맷 추가 후 기존 비율 계약 유지 |

### 3.2 Journey와 스토리보드

Journey는 문제 인식에서 회귀 보호까지 이어지도록 구성했다.

스토리보드는 다음 Stage를 사용했다.

| Stage |
|---|
| Awareness |
| Entry |
| Action |
| Validation |
| Outcome |

각 Stage에는 Action, Thinking, Emotion, Pain, Opportunity를 포함했다.

### 3.3 Story

User Story는 다음 6개로 구성했다.

| Story | 주제 |
|---|---|
| S1 | 입력 검증 |
| S2 | 레지스트리·OCP |
| S3 | 환산 정확도 |
| S4 | 출력 포맷 |
| S5 | 설정 로드 실패 |
| S6 | 동적 단위 등록 |

### 3.4 Gherkin

Gherkin Feature는 README 비율을 Background로 고정하고, 총 8개 Scenario를 포함하도록 정리했다.

필수 포함 항목은 다음과 같다.

| 필수 항목 | 반영 방식 |
|---|---|
| 소수 파싱 실패 | `Reject decimal parsing failure` |
| 음수 입력 정책 | 정책 문장을 먼저 고정한 뒤 `Reject negative input according to fixed policy` 작성 |
| 원 입력 단위/값 보존 | `Preserve original input unit and value in output` |
| 설정 파일 형식 오류 | `Reject invalid configuration file format` |

### 3.5 Level 5 체크리스트

Level 5는 다음 컬럼으로 구성했다.

| 컬럼 | 의미 |
|---|---|
| 영역 | 검증할 요구사항 범주 |
| 검증 방법 | 누가 무엇을 어떻게 확인하는지 |
| 통과 기준 | 무엇을 확인하면 통과인지 |
| 추적 ID | 연결되는 Story 번호 |

각 항목은 "누가 무엇을 어떻게 검증하면 통과인지"가 한 줄에서 드러나도록 작성했다.
