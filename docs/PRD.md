# PRD: 확장 가능한 Java 단위 변환 학습 시스템

## 1. 프로젝트 개요

### 1.1 한 줄 목적문 (What / Who / Why)

Java/클린 아키텍처 학습자가 `meter`, `feet`, `yard` 단위 변환 요구를 입력 검증, 비율 계약, 도메인 분리, 회귀 테스트 기준으로 학습할 수 있도록 PRD 수준의 제품 요구사항을 정의한다.

### 1.2 배경 및 문제 정의 (관찰 관점)

| 관찰 | 문제 정의 |
|---|---|
| 단위 변환 과제는 계산식보다 입력 형식, 오류 처리, 출력 표현, 확장 요구가 먼저 흔들릴 수 있다. | 구현 중심으로 접근하면 `unit:value` 입력 계약과 실패 조건이 테스트 가능한 기준으로 고정되지 않는다. |
| Phase 4 Epic은 README의 `meter`, `feet`, `yard` 비율을 기준 계약으로 고정했다. | `1 meter = 3.28084 feet`, `1 meter = 1.09361 yard`가 문서와 테스트의 공통 기준으로 남지 않으면 리팩터링 중 회귀를 식별할 수 없다. |
| Phase 4 Journey는 문제 인식, 계약 정의, 도메인 분리, Dual-Track TDD, 확장 요구 확인, 회귀 보호 순서로 학습 흐름을 정의했다. | PRD가 이 흐름을 제품 목표와 사용자 시나리오로 연결하지 않으면 학습자는 단위 변환 구현과 요구사항 분석을 구분하기 어렵다. |
| 새 단위, 출력 포맷, 외부 설정은 기존 변환 계약 위에 추가되는 확장 요구다. | 확장 요구의 경계를 정하지 않으면 기본 비율, 음수 정책, 미지원 단위 정책이 변경될 수 있다. |

### 1.3 목표

| ID | 목표 | 측정 기준 |
|---|---|---|
| G1 | Phase 4 Epic의 4개 목적을 PRD 개요와 사용자 시나리오에 반영한다. | PRD 1~2번 섹션에서 계약 정의, 테스트 설계, 도메인 분리, 확장성, 회귀 보호가 각각 1회 이상 명시되어야 한다. |
| G2 | README 기본 비율을 제품 기준 계약으로 고정한다. | `1 meter = 3.28084 feet`, `1 meter = 1.09361 yard`, `feet`↔`yard` meter 기준 계산이 PRD 1~2번 섹션에 명시되어야 한다. |
| G3 | Phase 4 Journey 기반 사용자 시나리오를 작성한다. | 문제 인식, 계약 정의, 도메인 분리, 검증, 회귀 보호 흐름을 포함하는 시나리오가 3개 이상 존재해야 한다. |
| G4 | 학습 성공을 테스트 가능한 요구사항으로 표현한다. | 입력 검증, 비율 정확도, 미지원 단위, 설정 실패, 원 입력 보존 중 4개 이상이 측정 가능한 학습 결과로 연결되어야 한다. |
| G5 | 확장 요구의 회귀 보호 기준을 정의한다. | 새 단위, 출력 포맷, 외부 설정이 추가되어도 기본 `meter`, `feet`, `yard` 계약을 변경하지 않는다는 문장이 포함되어야 한다. |

### 1.4 비목표(Non-Goal)

| ID | Non-Goal | 경계 |
|---|---|---|
| NG1 | 구현 산출물을 작성하지 않는다. | 코드, 클래스 설계, 메서드 시그니처, 패키지 구조, 빌드 스크립트는 PRD 범위에 포함하지 않는다. |
| NG2 | 저장소 범위 밖 제품 기능을 정의하지 않는다. | GUI, 웹 API, 데이터베이스, 인증, 배포, 네트워크 통신은 요구사항에 포함하지 않는다. |
| NG3 | 물리 단위 카탈로그를 확장하지 않는다. | 기본 계약은 `meter`, `feet`, `yard`로 제한하며, `cubit`은 동적 등록 예시로만 사용한다. |

### 3.2 기능별 입·출력 계약

| 기능 | 입력 문자열 계약 | 성공 출력 문자열 계약 | 실패 문자열 계약 |
|---|---|---|---|
| 입력 검증 | 변환 입력은 `unit:value` 형식의 문자열이어야 한다. 예: `meter:2.5` | 검증 성공 시 단위명 문자열 `meter`와 값 문자열 `2.5`가 원문 기준으로 보존되어야 한다. | `meter=2.5`, `meter:`, `:2.5`, `meter:2.5.1`, `meter:-1`은 변환 결과 없이 오류 문자열로 분류되어야 한다. |
| 기본 단위 변환 | 입력 단위 문자열은 `meter`, `feet`, `yard` 중 하나여야 한다. | `meter:2.5` 입력은 `feet`, `yard` 대상 결과 문자열을 생성하고 `meter` 대상 결과 문자열은 생성하지 않아야 한다. | `mile:1`은 미지원 단위 오류 문자열로 분류되고 변환 결과 문자열을 생성하지 않아야 한다. |
| README 비율 변환 | 변환 기준 문자열은 `1 meter = 3.28084 feet`, `1 meter = 1.09361 yard`로 고정한다. | `meter:1`은 `3.28084 feet`, `1.09361 yard` 의미의 결과를 생성해야 하며, 표시 반올림은 출력 포맷 정책을 따른다. | README 기준 비율이 아닌 값으로 생성된 결과는 계약 위반으로 판정되어야 한다. |
| meter 허브 교차 변환 | `feet:3.28084`와 `yard:1.09361`은 각각 `1 meter` 기준 입력과 같은 의미로 처리되어야 한다. | `feet`와 `yard` 사이의 결과는 meter 기준 환산을 거친 결과 문자열로 표현되어야 한다. | `feet`↔`yard` 직접 비율을 별도 기준으로 사용하는 결과는 계약 위반으로 판정되어야 한다. |
| 콘솔 출력 | 입력 문자열의 원 값과 원 단위를 출력에 사용해야 한다. | 각 행은 `<originalValue> <originalUnit> = <convertedValue> <targetUnit>` 형식을 따라야 한다. 예: `2.5 meter = 8.2 feet` | 입력 검증 실패, 미지원 단위, 설정 로드 실패 시 변환 결과 행을 출력하지 않아야 한다. |
| JSON 출력 | 출력 형식 요청 문자열이 `json`일 때 적용한다. | 성공 출력은 `input.value`, `input.unit`, `results[].unit`, `results[].value`, `results[].displayValue`, `error` 필드명을 포함해야 한다. | 실패 출력은 `results`가 비어 있음을 표현하고 `error` 필드에 오류 분류 문자열을 포함해야 한다. |
| CSV 출력 | 출력 형식 요청 문자열이 `csv`일 때 적용한다. | 첫 행은 `input_value,input_unit,target_unit,converted_value,display_value` 헤더 문자열이어야 하며 이후 행은 결과 문자열이어야 한다. | 실패 시 변환 결과 행을 생성하지 않고 오류 분류 문자열을 별도로 표현해야 한다. |
| 표 형태 출력 | 출력 형식 요청 문자열이 `table`일 때 적용한다. | 열 이름 문자열은 `Input Value`, `Input Unit`, `Target Unit`, `Converted Value`, `Ratio Source`를 포함해야 한다. | 실패 시 변환 결과 행을 생성하지 않고 오류 분류 문자열을 표시해야 한다. |
| 설정 로드 | 설정 입력은 JSON, YAML, 또는 메모리 기본값 중 하나의 문자열 기반 설정으로 제공될 수 있다. | 유효 설정은 단위명 문자열과 `metersPerUnit` 값 문자열을 제공해야 한다. | 형식 오류, 필수 단위명 누락, 필수 비율 누락, 숫자가 아닌 비율, 0 이하 비율은 설정 로드 실패 문자열로 분류되어야 한다. |
| 동적 단위 등록 | 등록 입력은 `register:<unit>:<metersPerUnit>` 형식이어야 한다. 예: `register:cubit:0.4572` | 등록 성공 후 `cubit`은 변환 입력 단위 문자열과 출력 대상 단위 문자열로 사용할 수 있어야 한다. | 빈 단위명, 숫자 파싱 실패, 0 이하 비율, 중복 단위명은 등록 실패 문자열로 분류되어야 한다. |

### 3.3 제약 사항

| 제약 | 계약 |
|---|---|
| 입력 형식 | 변환 입력은 콜론 1개를 포함하는 `unit:value` 문자열이어야 한다. |
| 빈 문자열 | `unit`과 `value`는 trim 후 빈 문자열이면 안 된다. |
| 허용 기본 단위 | Phase 4 Background Given과 동일하게 기본 등록 단위는 `meter`, `feet`, `yard`로 고정한다. |
| 비율 기준 | Phase 4 Background Given과 동일하게 `1 meter = 3.28084 feet`, `1 meter = 1.09361 yard`를 고정 비율로 사용한다. |
| 교차 변환 | Phase 4 Background Given과 동일하게 `feet`↔`yard`는 meter 기준 비율로 계산한다. |
| 숫자 값 | 값 문자열은 십진수로 파싱 가능해야 한다. |
| 0 값 정책 | `meter:0`, `feet:0`, `yard:0`은 유효 입력이며 모든 대상 단위의 정밀 변환 값은 `0`이어야 한다. |
| 음수 정책 | 0 미만 값은 유효하지 않은 입력이며 변환 결과 문자열을 생성하지 않는다. |
| 미지원 단위 | 동적 등록 전 `mile:1` 같은 등록되지 않은 단위는 미지원 단위로 거부한다. |
| 출력 대상 | 입력 단위와 동일한 단위는 변환 결과 대상에서 제외한다. |
| 원 입력 보존 | 모든 출력 포맷은 원 입력 값 문자열과 원 입력 단위 문자열을 변경 없이 포함해야 한다. |
| 중복 등록 | 이미 등록된 단위명은 동적 등록에서 거부한다. |

| SRP | 입력 검증, 변환 규칙, 단위 등록, 설정 로드, 출력 표현 책임은 서로 분리되어야 한다. |
| BCE 레이어 | Boundary는 외부 입력·출력 계약을 담당하고, Control은 흐름 조정을 담당하며, Entity는 단위·비율·변환 규칙을 담당한다. |
| 의존성 방향 | Boundary는 Control에 의존할 수 있고, Control은 Entity 규칙을 사용할 수 있으며, Entity는 Boundary 출력 형식에 의존하지 않는다. |
| 도메인 보호 | 변환 정확도와 단위 비율 규칙은 출력 포맷 변경으로 바뀌지 않아야 한다. |

### 4.3 테스트 커버리지 목표

| 영역 | 대상 | 목표 |
|---|---|---|
| Domain | 단위 비율, meter 허브 변환, 입력 단위 제외 규칙 | 라인 커버리지 85% 이상, 비율 계약별 테스트 1개 이상 |
| Boundary | 입력 문자열 검증, 출력 포맷 계약, 오류 메시지 분류 | 분기 커버리지 90% 이상 |
| Data | 설정 로드, 비율 값 검증, 동적 등록 계약 | 라인 커버리지 80% 이상, 설정 실패 유형별 테스트 1개 이상 |
| Regression | README 비율, 음수 정책, 소수 파싱 실패, 미지원 단위, 원 입력 보존 | 각 항목별 계약 테스트 1개 이상 |

### 4.4 확장성 원칙

| 원칙 | 요구사항 |
|---|---|
| 새 단위 추가 | 새 단위는 meter 기준 비율을 제공해야 하며 기존 `meter`, `feet`, `yard` 비율을 변경하지 않아야 한다. |
| 새 출력 포맷 추가 | 새 포맷은 변환 결과 데이터를 재표현해야 하며 계산 결과를 변경하지 않아야 한다. |
| 설정 외부화 | 외부 설정이 실패하면 잘못된 비율로 변환하지 않고 설정 로드 실패로 중단해야 한다. |
| 회귀 보호 | 확장 기능 추가 후에도 Phase 4 Gherkin Background 계약은 그대로 통과해야 한다. |

## 5. 데이터 요구사항

### 5.1 단위 비율 상수

| 단위 | README 고정 비율 | meter 허브 등록 의미 | 계약 |
|---|---:|---:|---|
| meter | `1 meter = 1 meter` | `1 meter = 1 meter` | 기준 단위이다. |
| feet | `1 meter = 3.28084 feet` | `1 feet = 1 / 3.28084 meter` | README 고정 비율이다. |
| yard | `1 meter = 1.09361 yard` | `1 yard = 1 / 1.09361 meter` | README 고정 비율이다. |

### 5.2 설정 외부화 방식

| 방식 | 요구사항 |
|---|---|
| JSON | 단위명과 `metersPerUnit` 값을 포함해야 하며 `metersPerUnit`은 `1 unit = n meter` 의미를 가진다. |
| YAML | 단위명과 `metersPerUnit` 값을 포함해야 하며 `metersPerUnit`은 `1 unit = n meter` 의미를 가진다. |
| 메모리 기본값 | 외부 설정을 사용하지 않는 경우 README 기본 비율을 사용해야 한다. |
| 실패 처리 | 형식 오류, 누락 필드, 숫자가 아닌 비율, 0 이하 비율은 설정 로드 실패로 처리한다. |

### 5.3 동적 단위 등록 계약

| 항목 | 계약 |
|---|---|
| 등록 형식 | `register:<unit>:<metersPerUnit>` |
| 예시 문자열 | `register:cubit:0.4572` |
| 의미 | `1 cubit = 0.4572 meter`로 등록한다. |
| 단위명 조건 | 단위명은 비어 있으면 안 된다. |
| 비율 조건 | `metersPerUnit`은 숫자로 파싱 가능하고 0보다 커야 한다. |
| 등록 성공 | 등록된 단위는 이후 변환 입력 단위와 출력 대상에 포함된다. |
| 중복 정책 | 이미 등록된 단위명은 거부한다. |

## 6. 출력 요구사항

### 6.1 콘솔 기본 포맷

| 항목 | 요구사항 |
|---|---|
| 행 구조 | `<originalValue> <originalUnit> = <convertedValue> <targetUnit>` |
| 필드 순서 | 원 입력 값, 원 입력 단위, 등호, 변환 값, 대상 단위 순서로 출력한다. |
| 원 입력 보존 | `meter:2.5` 입력의 출력 행은 `2.5 meter`를 포함해야 한다. |
| 대상 단위 | 입력 단위를 제외한 지원 단위만 출력한다. |
| 소수 자릿수 | 콘솔의 변환 값은 소수점 첫째 자리까지 반올림해 표시한다. |
| 예시 계약 | `meter:2.5` 입력은 `2.5 meter = 8.2 feet`, `2.5 meter = 2.7 yard` 형태의 행을 생성해야 한다. |

### 6.2 JSON 출력 스키마

| 필드 | 타입 | 필수 | 계약 |
|---|---|---|---|
| `input.value` | string | 예 | 원 입력 값을 문자열로 보존한다. |
| `input.unit` | string | 예 | 원 입력 단위를 문자열로 보존한다. |
| `results[].unit` | string | 예 | 대상 단위명을 포함한다. |
| `results[].value` | number | 예 | 반올림 전 정밀 변환 값을 숫자로 포함한다. |
| `results[].displayValue` | string | 예 | 표시용 반올림 값을 문자열로 포함한다. |
| `error` | string 또는 null | 예 | 성공 시 null, 실패 시 오류 분류명을 포함한다. |

### 6.3 CSV 출력 스키마

| 컬럼 순서 | 컬럼명 | 계약 |
|---:|---|---|
| 1 | `input_value` | 원 입력 값을 문자열로 보존한다. |
| 2 | `input_unit` | 원 입력 단위를 문자열로 보존한다. |
| 3 | `target_unit` | 대상 단위명을 포함한다. |

| 5 | `display_value` | 콘솔 표시와 동일한 소수 자릿수 정책을 적용한다. |

### 6.4 표(Table) 출력 스키마

| 컬럼 순서 | 컬럼명 | 계약 |
|---:|---|---|
| 1 | Input Value | 원 입력 값을 표시한다. |
| 2 | Input Unit | 원 입력 단위를 표시한다. |
| 3 | Target Unit | 대상 단위를 표시한다. |

## 7. 성공 지표

### 7.1 인수 기준

| 체크 | 인수 기준 | Story 일치 |
|---|---|---|
| [ ] | `unit:value` 형식이 아닌 입력은 거부되고 변환 결과가 생성되지 않는다. | S1 |
| [ ] | `meter`, `feet`, `yard`는 기본 등록 단위로 식별된다. | S2 |
| [ ] | `1 meter = 3.28084 feet`, `1 meter = 1.09361 yard` 기준 변환이 절대 오차 `0.00001` 이하로 검증된다. | S3 |
| [ ] | 출력은 원 입력 단위와 원 입력 값을 보존한다. | S4 |
| [ ] | 설정 파일 형식 오류가 발생하면 설정 로드 실패로 보고되고 변환이 진행되지 않는다. | S5 |
| [ ] | `register:cubit:0.4572` 등록 후 `cubit`은 변환 입력 단위와 출력 대상에 포함된다. | S6 |

### 7.2 회귀 보호 규칙

| 규칙 | 계약 변경 금지 정책 |
|---|---|
| RR1 | README 기본 비율 `1 meter = 3.28084 feet`, `1 meter = 1.09361 yard`는 PRD 변경 없이 수정할 수 없다. |
| RR2 | feet↔yard 변환은 meter 허브 기준 계산이라는 Background Given을 유지해야 한다. |
| RR3 | 음수 입력 거부 정책은 PRD 변경 없이 허용 정책으로 바꿀 수 없다. |
| RR4 | 소수 파싱 실패 입력은 변환 결과를 생성하면 안 된다. |
| RR5 | 미지원 단위는 동적 등록 전까지 변환 입력으로 허용하면 안 된다. |
| RR6 | 출력의 원 입력 값과 원 입력 단위 보존 계약은 모든 출력 포맷에서 유지해야 한다. |
| RR7 | 결함 수정 시 해당 결함을 재현하는 계약 테스트가 먼저 정의되어야 한다. |

## 8. 용어 정의

| 용어 | 정의 |
|---|---|
| Unit | 변환 대상 길이 단위이며 기본값은 `meter`, `feet`, `yard`이다. |
| Meter Hub | 모든 단위 비율을 meter 기준으로 연결하는 기준 모델이다. |
| Ratio Contract | README에 고정된 단위 간 비율이며 테스트로 검증되어야 하는 계약이다. |
| Input Contract | 사용자가 제공하는 문자열 형식과 유효성 조건이다. |
| Output Contract | 변환 결과가 포함해야 하는 필드, 순서, 원 입력 보존 규칙이다. |
| Unsupported Unit | 기본 등록 단위 또는 동적 등록 단위에 포함되지 않은 단위이다. |
| Dynamic Registration | 실행 중 `register:<unit>:<metersPerUnit>` 형식으로 새 단위를 추가하는 요구사항이다. |
| Regression Protection | 기존 README 비율과 오류 정책이 기능 확장 후에도 깨지지 않도록 테스트로 보호하는 규칙이다. |
| Boundary | 외부 입력 검증과 출력 표현 책임을 갖는 계층 개념이다. |
| Control | 입력, 변환, 출력 흐름을 조정하는 계층 개념이다. |
| Entity | 단위, 비율, 변환 규칙 같은 도메인 계약을 담는 계층 개념이다. |
