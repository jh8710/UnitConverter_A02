# Unit Converter Requirements Package

이 문서는 `UnitConverter_A02` 저장소의 README 요구사항을 기준으로 작성한 요구사항 서술 패키지이다. 구현, 클래스 설계, 메서드 시그니처, 패키지 구조, 코드 예시는 포함하지 않는다.

## Level 1 Epic

| 항목 | 내용 |
|---|---|
| 제목 | 확장 가능한 Java 단위 변환 학습 시스템 |

### 목적

| No. | 목적 |
|---:|---|
| 1 | Java/클린 아키텍처 학습자가 단위 변환 문제를 통해 입력 검증, 계약 정의, 도메인 분리, 테스트 설계를 함께 학습한다. |
| 2 | README에 명시된 `meter`, `feet`, `yard` 비율을 기준 계약으로 고정하여 변환 정확도를 검증 가능하게 만든다. |
| 3 | 새 단위, 출력 포맷, 외부 설정이 추가되어도 기존 변환 계약을 깨뜨리지 않는 확장성을 요구사항으로 정의한다. |
| 4 | Dual-Track TDD 관점에서 기능 인수 기준과 설계 회귀 보호 기준을 함께 추적한다. |

### 성공 기준

| 기준 | 측정 방법 | 성공 목표 |
|---|---|---|
| 커버리지 | 테스트 커버리지 리포트 | 핵심 도메인 변환 로직 라인 커버리지 85% 이상, 입력 검증 분기 커버리지 90% 이상 |
| 계약 테스트 | README 비율, 입력 오류, unknown unit, 설정 오류에 대한 계약 테스트 확인 | 필수 계약별 최소 1개 이상의 테스트가 존재하고 통과 |
| 회귀 정책 | 결함 수정 및 확장 요구 반영 시 테스트 추가 여부 확인 | 모든 회귀 결함은 실패 재현 테스트를 먼저 정의한 뒤 수정 |
| 확장성 | 새 단위 또는 출력 포맷 추가 시 기존 계약 테스트 재실행 | `meter`↔`feet`↔`yard` 기존 비율 계약이 계속 통과 |

## Level 3 User Stories

### Story 1. 입력 검증

**As a** Java/클린 아키텍처 학습자  
**I want** `단위:값` 입력을 검증하고 잘못된 입력을 명확히 거부하고 싶다  
**So that** 변환 로직이 유효한 값만 처리하도록 보호할 수 있다.

Acceptance Criteria

- [ ] 입력은 정확히 `unit:value` 형식을 따라야 한다.
- [ ] 단위명과 값은 모두 비어 있으면 안 된다.
- [ ] 값은 숫자로 파싱 가능해야 한다.
- [ ] 음수 값은 유효하지 않은 입력으로 거부되어야 한다.
- [ ] 검증 실패 시 변환 결과는 생성되지 않아야 한다.

### Story 2. 레지스트리·OCP

**As a** Java/클린 아키텍처 학습자  
**I want** 지원 단위를 레지스트리 관점으로 관리하고 싶다  
**So that** 새 단위 추가 시 기존 변환 계약 변경을 최소화할 수 있다.

Acceptance Criteria

- [ ] 기본 등록 단위는 `meter`, `feet`, `yard`여야 한다.
- [ ] 지원 단위 여부는 고정 조건문이 아니라 등록된 단위 목록 기준으로 판단되어야 한다.
- [ ] 새 단위가 추가되어도 README 기본 비율 계약은 변경되지 않아야 한다.
- [ ] 입력 단위를 제외한 등록 단위가 변환 결과 대상이 되어야 한다.

### Story 3. 환산 정확도

**As a** Java/클린 아키텍처 학습자  
**I want** README 기준 비율로 단위를 정확히 환산하고 싶다  
**So that** 계산 결과를 신뢰 가능한 도메인 계약으로 검증할 수 있다.

Acceptance Criteria

- [ ] `1 meter = 3.28084 feet` 비율을 기준으로 사용해야 한다.
- [ ] `1 meter = 1.09361 yard` 비율을 기준으로 사용해야 한다.
- [ ] `feet`↔`yard` 변환은 `meter` 기준 비율을 통해 계산되어야 한다.
- [ ] 변환 결과는 합의된 허용 오차 이내여야 한다.
- [ ] 입력 단위와 동일한 단위의 변환 결과는 출력 대상에서 제외되어야 한다.

### Story 4. 출력 포맷

**As a** Java/클린 아키텍처 학습자  
**I want** 변환 결과를 선택한 출력 포맷으로 표현하고 싶다  
**So that** 동일한 변환 결과를 텍스트, JSON, CSV, 표 형태로 비교할 수 있다.

Acceptance Criteria

- [ ] 기본 출력은 `<originalValue> <originalUnit> = <convertedValue> <targetUnit>` 의미를 보존해야 한다.
- [ ] 출력은 원 입력 값과 원 입력 단위를 보존해야 한다.
- [ ] JSON 출력은 입력 정보와 변환 결과 목록을 구조적으로 표현해야 한다.
- [ ] CSV 출력은 헤더와 결과 행을 구분해야 한다.
- [ ] 표 형태 출력은 입력 단위, 대상 단위, 변환 값을 열로 구분해야 한다.
- [ ] 출력 포맷 변경은 변환 계산 결과를 변경하지 않아야 한다.

### Story 5. 설정 로드 실패

**As a** Java/클린 아키텍처 학습자  
**I want** 외부 설정 파일 오류를 명확히 감지하고 싶다  
**So that** 잘못된 비율 설정으로 변환 결과가 조용히 오염되지 않도록 보호할 수 있다.

Acceptance Criteria

- [ ] 설정 파일 형식이 올바르지 않으면 설정 로드 실패로 분류되어야 한다.
- [ ] 필수 단위명 또는 비율 값이 누락되면 설정 로드 실패로 분류되어야 한다.
- [ ] 숫자가 아닌 비율 값은 거부되어야 한다.
- [ ] 0 이하 비율 값은 거부되어야 한다.
- [ ] 설정 로드 실패 시 단위 변환은 진행되지 않아야 한다.
- [ ] 설정 로드 실패는 사용자 입력 검증 실패와 구분 가능해야 한다.

### Story 6. 동적 단위 등록

**As a** Java/클린 아키텍처 학습자  
**I want** 실행 중 새 단위와 `meter` 기준 비율을 등록하고 싶다  
**So that** 기존 기본 단위 계약을 유지하면서 변환 범위를 확장할 수 있다.

Acceptance Criteria

- [ ] `1 cubit = 0.4572 meter` 같은 등록 요구를 표현할 수 있어야 한다.
- [ ] 등록 단위명은 비어 있으면 안 된다.
- [ ] 등록 비율은 숫자로 파싱 가능하고 0보다 커야 한다.
- [ ] 등록된 단위는 이후 변환 입력 단위로 사용할 수 있어야 한다.
- [ ] 등록된 단위는 기존 단위 변환의 출력 대상에 포함되어야 한다.
- [ ] 중복 단위명 등록 정책은 거부 또는 명시적 갱신 중 하나로 고정되어야 한다.

## Level 4 Gherkin Feature

**음수 입력 정책:** 길이 단위 변환에서 0 미만 값은 유효하지 않은 입력이며, 변환 결과를 생성하지 않는다.

```gherkin
Feature: README 기준 단위 변환 계약
  Java/클린 아키텍처 학습자로서
  README에 고정된 meter, feet, yard 비율을 기준으로
  입력 검증, 환산 정확도, 출력 표현, 설정 실패를 회귀 보호하고 싶다.

  Background:
    Given README 기준 비율은 다음과 같이 고정되어 있다
      | from  | to    | ratio   |
      | meter | feet  | 3.28084 |
      | meter | yard  | 1.09361 |
    And feet와 yard 간 변환은 meter 기준 비율로 계산한다
    And 기본 등록 단위는 meter, feet, yard 이다

  Scenario: Convert meter input to feet and yard
    Given 사용자가 "meter:2.5"를 입력한다
    When 단위 변환을 요청한다
    Then feet 변환 결과가 생성되어야 한다
    And yard 변환 결과가 생성되어야 한다
    And meter 변환 결과는 생성되지 않아야 한다
    And 변환 결과는 README 기준 비율의 허용 오차 이내여야 한다

  Scenario: Convert feet and yard input through meter baseline
    Given 사용자가 "feet:3.28084"를 입력한다
    And 사용자가 "yard:1.09361"를 입력한다
    When 단위 변환을 요청한다
    Then feet 입력의 meter 변환 결과는 "1 meter" 기준과 일치해야 한다
    And yard 입력의 meter 변환 결과는 "1 meter" 기준과 일치해야 한다
    And feet와 yard 간 변환 결과는 meter 기준 비율을 통해 계산되어야 한다

  Scenario: Reject malformed input without colon separator
    Given 사용자가 "meter=2.5"를 입력한다
    When 입력 검증을 수행한다
    Then 입력은 잘못된 형식으로 거부되어야 한다
    And 변환 결과는 생성되지 않아야 한다

  Scenario: Reject decimal parsing failure
    Given 사용자가 "meter:2.5.1"를 입력한다
    When 입력 검증을 수행한다
    Then 입력은 숫자 파싱 실패로 거부되어야 한다
    And 변환 결과는 생성되지 않아야 한다

  Scenario: Reject negative input according to fixed policy
    Given 음수 입력 정책은 0 미만 값을 거부하도록 고정되어 있다
    And 사용자가 "meter:-1"를 입력한다
    When 입력 검증을 수행한다
    Then 입력은 음수 값으로 거부되어야 한다
    And 변환 결과는 생성되지 않아야 한다

  Scenario: Reject unknown unit
    Given 사용자가 "mile:1"을 입력한다
    When 단위 변환을 요청한다
    Then 등록되지 않은 단위 오류가 발생해야 한다
    And 변환 결과는 생성되지 않아야 한다

  Scenario: Preserve original input unit and value in output
    Given 사용자가 "meter:2.5"를 입력한다
    When 변환 결과를 출력한다
    Then 출력은 원 입력 값 "2.5"를 보존해야 한다
    And 출력은 원 입력 단위 "meter"를 보존해야 한다
    And 각 변환 결과는 "2.5 meter"에서 파생된 표현이어야 한다

  Scenario: Reject invalid configuration file format
    Given 외부 설정 파일의 형식이 올바르지 않다
```
