# 22_최종 회귀 확인 및 PR 보고서

## 1. 작업 개요

이번 세션에서는 `UnitConverter_A02` 프로젝트의 최종 회귀 상태를 확인했다.

사용자가 요청한 확인 범위는 다음과 같았다.

- 전체 테스트 실행: `mvn test`
- 커버리지 리포트 생성: `mvn jacoco:report`
- 모든 TC 통과 여부 확인: TC-A-01~07, TC-B-01~07
- Golden Master 테스트 통과 여부 확인
- if-else 체인 제거 및 `UnitRegistry` 교체 여부 확인
- 변환 비율 `3.28084`, `1.09361` 인라인 잔존 여부 확인
- Domain(변환 로직)과 Boundary(파싱, 출력) 분리 여부 확인
- Domain 커버리지 95% 이상, Boundary 커버리지 85% 이상 확인

이번 세션에서는 코드 수정이나 PR 생성은 수행하지 않고, 현재 리포지토리 상태를 기준으로 회귀 테스트와 구조 조건을 점검했다.

확인 결과 이 프로젝트는 Maven 프로젝트이며, 루트에 `pom.xml`이 존재한다.

Gradle 빌드 파일은 확인되지 않았다.

## 6. 구조 조건 확인

변환 비율 인라인 잔존 여부를 확인했다.

```shell
rg "3\.28084|1\.09361" --glob "*.java"
```

프로덕션 코드 기준으로 다음 위치에 값이 남아 있었다.

- `src/main/java/UnitConverter.java`
- `src/main/java/com/example/unitconverter/UnitConverterService.java`
- 루트 `UnitConverter.java`

`UnitConverterService.java`에는 상수로 정의되어 있었다.

```java
public static final double FEET_PER_METER = 3.28084;
public static final double YARD_PER_METER = 1.09361;
```

반면 루트 `UnitConverter.java`에는 변환식에 매직 넘버가 직접 인라인으로 남아 있었다.

```java
meterValue = value / 3.28084;
meterValue = value / 1.09361;
double inFeet = meterValue * 3.28084;
double inYards = meterValue * 1.09361;
```

if-else 체인 및 `UnitRegistry` 교체 여부도 확인했다.

`UnitRegistry` 클래스, 레코드, 인터페이스는 검색되지 않았다.

또한 루트 `UnitConverter.java`에는 다음과 같은 if-else 체인이 남아 있었다.

```java
if (unit.equals("meter")) {
    meterValue = value;
} else if (unit.equals("feet")) {
    meterValue = value / 3.28084;
} else if (unit.equals("yard")) {
    meterValue = value / 1.09361;
}
```

따라서 if-else 체인 제거 및 `UnitRegistry` 교체 조건은 충족되지 않았다.

## 7. Domain / Boundary 분리 확인

현재 `src/main/java/com/example/unitconverter/UnitConverterService.java` 안에는 다음 책임이 함께 존재한다.

- 단위 등록 및 변환 비율 관리
- 변환 계산
- 입력 문자열 파싱
- 콘솔 출력 라인 포맷팅
- JSON/YAML 설정 파싱

별도 `domain`, `boundary`, `parser`, `formatter`, `UnitRegistry` 구조는 확인되지 않았다.

따라서 Domain(변환 로직)과 Boundary(파싱, 출력)가 명확히 분리되었다고 판정할 수 없다.

또한 패키지나 클래스 단위로 Domain과 Boundary가 분리되어 있지 않아, Domain 커버리지 95% 이상 및 Boundary 커버리지 85% 이상을 독립 수치로 산정할 수 없었다.
