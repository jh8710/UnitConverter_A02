# 03. .cursorrules 다단계 보고서

## 1. 목적

이번 세션의 목적은 `UnitConverter_A02` 프로젝트 루트에 둘 `.cursorrules` 규칙을 단계적으로 설계하고, Java 단위 변환기 개발에 필요한 코드 스타일, BCE 아키텍처, TDD 규칙, 테스트 기준, 금지 패턴, 파일 구조, AI 행동 규칙을 완성 YAML 형태로 정리하는 것이다.

대상 프로젝트는 Java 17 이상을 가정한 단위 변환기이며, Google Java Format, JUnit 5, AAA 테스트 구조, Boundary-Control-Entity 계층 분리, RED-GREEN-REFACTOR 흐름을 기본 개발 규칙으로 삼는다.

## 2. 진행 단계

### 2.1 YAML 뼈대 작성

초기 요청에서는 `.cursorrules`의 최상위 키만 가진 YAML 뼈대를 구성했다.

- `project`
- `code_style`
- `architecture`
- `tdd_rules`
- `testing`
- `forbidden`
- `file_structure`
- `ai_behavior`

각 최상위 키 앞에는 80자 폭의 `#` 구분선 주석을 배치하는 형식을 사용했다.

### 2.2 `tdd_rules` 블록 작성

두 번째 단계에서는 Java 단위 변환기 기준으로 `tdd_rules`만 채웠다.

- `red_phase`: 실패하는 JUnit 5 테스트를 먼저 작성하고, 변환 계약과 입력 검증을 테스트로 고정한다.
- `green_phase`: 실패한 테스트를 통과시키는 최소 Java 구현만 허용한다.
- `refactor_phase`: 테스트가 통과한 뒤에만 이름, 중복, 책임 경계를 개선한다.

각 phase는 `description`, `rules`, `must_not` 구조를 가진다.

### 2.3 완성본 검토

세 번째 단계에서는 실제 `.cursorrules` 전체를 읽고 다음 항목만 점검했다.

- YAML 문법 오류 가능성
- 누락된 필수 섹션
- `tdd_rules`와 `forbidden` 충돌 여부
- Cursor가 지키기 어려운 `ai_behavior` 규칙

검토 결과, 필수 섹션은 모두 존재했고 `tdd_rules`와 `forbidden` 사이의 뚜렷한 충돌은 없었다. 다만 `ai_behavior.must_not` 끝의 빈 리스트 항목은 YAML 해석상 빈 항목으로 처리될 가능성이 있었다.

### 2.4 최종 YAML 완성

마지막 단계에서는 프로젝트 루트 `.cursorrules`에 넣을 완성 YAML을 새로 구성했다.

완성 YAML은 다음 원칙을 반영한다.

- Java 17 이상을 기준으로 한다.
- Google Java Format을 고정한다.
- Boundary, Control, Entity의 책임과 금지 책임을 분리한다.
- 의존성 방향은 `boundary -> control -> entity`로 제한한다.
- 테스트 프레임워크는 JUnit 5로 고정한다.
- 테스트 구조는 AAA를 사용한다.
- Fixture는 테스트 가독성을 해치지 않는 범위에서만 공유한다.
- 커버리지 기준은 Entity 95, Control 90, Boundary 85, Overall 90으로 둔다.
- Java 특화 금지 패턴을 `pattern`, `reason`, `alternative`로 기록한다.
- AI는 레이어 위반 금지, 테스트 먼저 작성, green 이후 리팩터를 따라야 한다.

## 3. 완성 `.cursorrules` YAML

```yaml
project:
  language: Java
  java_version: "17+"
  purpose: Unit converter implemented with TDD and clear layer separation.

code_style:
  formatter: Google Java Format
  naming:
    packages: lower.case.package.names
    classes: UpperCamelCase
    interfaces: UpperCamelCase
    methods: lowerCamelCase
    fields: lowerCamelCase
    constants: UPPER_SNAKE_CASE
    test_classes: ClassNameTest
    test_methods: shouldDescribeExpectedBehavior
  packages:
    boundary: Handles input parsing, request validation, output formatting, and external adapters.
    control: Coordinates application use cases and layer interaction.
    entity: Contains domain models, invariants, conversion rules, and pure business logic.
  modules:
    boundary_must_not_contain:
      - unit conversion formulas
      - domain invariants
      - persistence details
    control_must_not_contain:
      - input parsing details
      - output serialization details
      - raw conversion constants
    entity_must_not_contain:
      - console input or output
      - file system access
      - framework-specific annotations

architecture:
  style: boundary-control-entity
  boundary:
    responsibility:
      - Parse external input.
      - Validate request shape.
      - Convert control results into user-facing output.
      - Present errors without owning business rules.
    must_not:
      - Perform unit conversion formulas.
      - Own conversion ratios.
      - Depend directly on entity internals beyond public contracts.
  control:
    responsibility:
      - Orchestrate one application use case.
      - Call entity services using explicit inputs and outputs.
      - Coordinate boundary requests with domain behavior.
    must_not:
      - Contain domain invariants.
      - Serialize output formats.
      - Hide invalid input with silent defaults.
  entity:
    responsibility:
      - Represent units, quantities, conversion ratios, and conversion results.
      - Enforce domain invariants.
      - Perform deterministic unit conversion.
    must_not:
      - Depend on boundary classes.
      - Depend on control classes.
      - Read files, write console output, or parse external formats.
  dependency_direction:
    allowed:
      - boundary -> control
      - control -> entity
    forbidden:
      - entity -> control
      - entity -> boundary
      - control -> boundary

tdd_rules:
  red_phase:
    description: Write a failing JUnit 5 test before changing production behavior.
    rules:
      - Test one observable conversion contract at a time.
      - Assert expected numeric results with an explicit precision policy.
      - Cover normal input, boundary values, and invalid input.
    must_not:
      - Do not write production behavior before a failing test exists.
      - Do not weaken assertions to make a test pass.
      - Do not test private implementation details.
  green_phase:
    description: Implement the smallest behavior needed to pass the failing test.
    rules:
      - Keep the implementation direct and minimal.
      - Preserve all previously passing tests.
      - Keep conversion formulas consistent with registered unit ratios.
    must_not:
      - Do not add untested features.
      - Do not introduce unnecessary abstractions.
      - Do not change expected test values to match incorrect implementation.
  refactor_phase:
    description: Improve structure only after the relevant tests are green.
    rules:
      - Remove duplication without changing behavior.
      - Improve names and layer boundaries.
      - Rerun affected JUnit 5 tests after meaningful refactoring.
    must_not:
      - Do not introduce new behavior during refactoring.
      - Do not merge boundary, control, and entity responsibilities.
      - Do not refactor while tests are failing.

testing:
  framework: JUnit 5
  structure: AAA
  aaa:
    arrange: Create explicit inputs, units, converters, and fixtures.
    act: Execute one public method or use case.
    assert: Verify observable results, errors, precision, and output shape.
  fixtures:
    rules:
      - Keep fixtures local to the test class unless reused by multiple test classes.
      - Name fixtures after the behavior they support.
      - Do not hide important input values inside shared setup.
      - Prefer in-memory fixtures for unit tests.
      - Use file fixtures only for integration or parser tests.
  coverage:
    entity: 95
    control: 90
    boundary: 85
    overall: 90
  required_test_types:
    - entity_unit_tests
    - control_use_case_tests
    - boundary_contract_tests
    - integration_tests

forbidden:
  - name: excessive_system_out_debugging
    pattern: System.out.println or System.err.println used as temporary debugging in production code.
    reason: Console debugging pollutes user output and hides behavior from tests.
    alternative: Use JUnit assertions for verification and a logger for intentional runtime diagnostics.
  - name: hardcoded_magic_numbers
    pattern: Raw conversion ratios, precision values, or rounding scales repeated inside methods.
    reason: Repeated numeric literals make conversion changes unsafe and obscure domain meaning.
    alternative: Use named constants, unit definitions, or domain value objects.
  - name: ignored_broad_exception
    pattern: catch (Exception e) with an empty body, comment-only body, or silent fallback.
    reason: Broad ignored exceptions hide parsing, validation, and conversion failures.
    alternative: Catch specific exceptions and convert them to tested error results.
  - name: public_field_abuse
    pattern: public mutable fields on boundary, control, or entity classes.
    reason: Public mutable state bypasses invariants and makes tests order-dependent.
    alternative: Use private final fields, constructor validation, and explicit accessors when needed.
  - name: god_class
    pattern: One class parses input, converts units, validates data, formats output, and handles errors.
    reason: Mixed responsibilities violate layer separation and make regression testing brittle.
    alternative: Split behavior into boundary, control, and entity classes.
  - name: static_global_mutable_registry
    pattern: static mutable Map, List, or Set used as the shared unit registry.
    reason: Global mutable state leaks between JUnit 5 tests and breaks deterministic behavior.
    alternative: Inject registry instances or use immutable unit definitions.
  - name: floating_point_equality_without_policy
    pattern: Direct equality assertions or comparisons on double conversion results.
    reason: Floating-point conversion requires an explicit precision and rounding policy.
    alternative: Use BigDecimal where appropriate or assert with a defined tolerance.

file_structure:
  tree:
    - path: src/main/java/.../boundary/
      comment: Input adapters, request validation, presenters, and output formatting.
    - path: src/main/java/.../control/
      comment: Application use cases and orchestration services.
    - path: src/main/java/.../entity/
      comment: Domain models, conversion rules, invariants, and pure business logic.
    - path: src/test/java/.../boundary/
      comment: Boundary contract tests.
    - path: src/test/java/.../control/
      comment: Use case and orchestration tests.
    - path: src/test/java/.../entity/
      comment: Domain unit tests.
    - path: src/test/resources/
      comment: Test fixtures for parser, boundary, and integration tests.

ai_behavior:
  rules:
    - Do not violate the declared dependency direction.
    - Write or update JUnit 5 tests before changing production behavior.
    - Keep refactoring work after the relevant tests are green.
    - Preserve existing public contracts unless a failing test intentionally changes them.
    - Keep boundary, control, and entity responsibilities separate.
    - Prefer small changes that protect one observable behavior at a time.
  must_not:
    - Do not add conversion behavior without a failing test first.
    - Do not place parsing, conversion, validation, formatting, and orchestration in one class.
    - Do not introduce new frameworks without explicit project need.
    - Do not replace precise assertions with weaker assertions.
```

## 4. 검토 결과

- YAML 최상위 필수 섹션은 모두 포함했다.
- Java 단위 변환기 기준의 TDD 흐름을 `red_phase`, `green_phase`, `refactor_phase`로 분리했다.
- `forbidden`은 Java 특화 금지 패턴을 7개로 확장했다.
- `ai_behavior`는 Cursor가 따라야 할 개발 규칙을 테스트 우선, 레이어 분리, green 이후 리팩터 중심으로 정리했다.
