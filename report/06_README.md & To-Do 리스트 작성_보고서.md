| 산출물 | 경로 | 목적 |

## 2. README 작성 내용

`docs/README.md`는 Phase 5 PRD의 목적문과 계약을 바탕으로 오픈소스 문서 형식으로 재구성했다.

반영한 주요 섹션은 다음과 같다.

- 프로젝트 이름과 한 줄 설명

- 개요 및 주요 학습 목표
- PRD 문서 링크
- 빠른 시작과 예시 입출력
- 지원 단위 및 meter 기준 비율 표
- 입력 형식 계약과 오류 케이스
- BCE 아키텍처 다이어그램
- 새 단위 추가 방법
- 테스트 실행과 커버리지 목표
- JSON/YAML 설정 예시
- Console, JSON, CSV 출력 예시
- 기여 가이드
- MIT 라이선스

## 4. README와 PRD 비교 결과

완성된 `docs/README.md`와 `docs/PRD.md`를 비교해 다음 범주로 보고했다.

- PRD에 있으나 README에 빠진 항목
- README에 있으나 PRD에 없는 범위 초과 위험 항목
- 입출력 계약 불일치
- 테스트 커버리지 목표 불일치

주요 확인 결과는 다음과 같다.

- README의 커버리지 목표는 PRD 4.3과 일치한다.
- README에는 PRD 6.4의 Table 출력 예시가 본문 출력 포맷에 부족하다.
- README의 오류 메시지 패턴은 PRD에 구체 문자열로 정의되어 있지 않아 범위 초과 가능성이 있다.
- README의 Quick Start 실행 명령과 설정 파일 경로는 PRD에 구체 명령·경로로 정의되어 있지 않다.

## 5. Phase 4 Story와 To-Do 추적

Phase 4 `docs/requirements.md`의 User Stories와 Phase 6 To-Do 리스트를 대조해 추적 매트릭스를 작성했다.

대조한 Story는 다음과 같다.

| Story ID | Story 제목 | 추적 상태 |
|---|---|---|
| S1 | 입력 검증 | 대응 To-Do 있음 |
| S2 | 레지스트리·OCP | 부분 대응 |
| S3 | 환산 정확도 | 대응 To-Do 있음 |
| S4 | 출력 포맷 | 대응 To-Do 있음 |
| S5 | 설정 로드 실패 | 대응 To-Do 있음 |
| S6 | 동적 단위 등록 | 부분 대응 |

확인한 누락 To-Do는 다음과 같다.

- S2의 “지원 단위 여부는 고정 조건문이 아니라 등록된 단위 목록 기준으로 판단”에 대한 명시적 To-Do가 부족하다.
- S6의 “등록 단위명은 비어 있으면 안 된다”에 대한 동적 등록 전용 To-Do가 부족하다.
- S6의 “등록 비율은 숫자로 파싱 가능하고 0보다 커야 한다”에 대한 동적 등록 전용 To-Do가 부족하다.
- S6의 중복 단위명 정책은 Phase 4에서는 “거부 또는 명시적 갱신”이지만 To-Do와 Phase 5 PRD에서는 “거부”로 고정되어 있다.

확인한 고아 To-Do는 다음과 같다.

- 실제 빌드 도구 선택 전 문서 명령이 병렬로 남아 있다는 기술 부채 항목
- Done 항목의 문서화 완료 기록
- 마일스톤 목표일
