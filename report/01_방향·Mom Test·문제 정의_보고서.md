# 01. 방향·Mom Test·문제 정의 보고서

## 1. 진행 방향

이번 세션에서는 Unit Converter 프로젝트를 구현 과제가 아니라 학습자의 행동과 사고를 드러내는 문제 정의 과제로 다루었다. 핵심 방향은 길이 단위 변환 기능 자체보다, 입력 검증, 변환 비율 계약, 출력 표현, 설정 실패, 동적 단위 등록, 회귀 보호를 학습 가능한 요구사항으로 정리하는 것이다.

이 프로젝트는 meter, feet, yard 변환이라는 작고 익숙한 도메인을 사용한다. 계산 난도는 낮추고, 대신 OCP/SRP, 계약 테스트, 책임 분리, 확장 요구의 위험을 관찰할 수 있도록 설계된 학습 맥락이다.

## 2. Observation

현재 상황은 단순한 길이 환산기를 만드는 문제가 아니라, meter, feet, yard 변환이라는 작은 도메인을 통해 “무엇이 유효한 입력인가”, “어떤 비율이 기준 계약인가”, “결과는 어떤 형식으로 보장되어야 하는가”, “확장 시 기존 동작은 어떻게 보호되는가”를 먼저 명확히 하려는 학습 상황이다. 계산 자체보다 입력 검증, 변환 정확도, 출력 표현, 설정 실패, 새 단위 추가가 서로 섞이지 않도록 문제를 정의하는 것이 핵심이다.

길이 변환은 누구나 이해하기 쉬운 문제이면서도, 실제로는 정확한 비율, 단위 목록, 잘못된 입력, 반올림, 출력 형식, 확장 요구를 모두 포함한다. 따라서 알고리즘 난도를 낮추고, 대신 계약 정의와 테스트 가능성, 책임 분리 같은 설계 사고에 집중하기 좋다.

이 문제는 OCP/SRP를 추상 개념이 아니라 측정 가능한 요구사항으로 바꾸는 연습이다. 입력 검증은 변환 로직을 오염시키지 않기 위한 경계 문제이고, 설정 외부화와 동적 단위 등록은 새 요구사항이 들어와도 기존 변환 계약이 깨지지 않아야 한다는 OCP 학습 맥락이다. 출력 포맷은 같은 변환 결과를 여러 표현으로 바꾸되 계산 규칙은 바뀌지 않아야 하므로, 도메인 책임과 표현 책임을 분리하는 SRP 학습으로 연결된다.

## 3. Why

### Why #1. 정확한 변환 결과가 왜 중요한가?

정확한 변환 결과는 이 시스템의 신뢰 기준이기 때문이다. meter, feet, yard의 기준 비율이 흔들리면 정상 입력과 정상 출력의 의미가 사라지고, 이후 입력 검증이나 출력 포맷이 아무리 잘 되어 있어도 사용자는 결과를 믿을 수 없다.

또한 정확도는 회귀 보호의 기준점이다. 새 단위 등록, 설정 외부화, 출력 포맷 추가 같은 확장이 들어와도 기본 비율이 유지되는지 확인할 수 있어야 한다. 즉 정확한 변환 결과는 기능 하나가 아니라 전체 설계가 지켜야 할 핵심 계약이다.

### Why #2. 수동 계산이 아닌 프로그램과 테스트인 이유

수동 계산은 한 번의 답을 확인할 수는 있지만, 반복되는 입력과 예외 상황, 확장 이후의 회귀를 안정적으로 검증하지 못한다. 프로그램은 동일한 규칙을 일관되게 적용하고, 테스트는 그 규칙이 시간이 지나도 깨지지 않도록 고정한다.

특히 이 문제는 정상 변환만이 아니라 잘못된 형식, 숫자 파싱 실패, 음수 입력, 미지원 단위, 설정 오류, 출력 원본 보존까지 다룬다. 이런 조건들은 사람이 매번 확인하기 어렵기 때문에 테스트 가능한 계약으로 전환해야 한다.

### Why #3. 계약을 먼저 고정하려는 이유

계약을 먼저 고정하지 않으면 구현 과정에서 “올바른 동작”의 기준이 계속 바뀐다. 그러면 변환 계산, 입력 검증, 출력 형식, 단위 등록 정책이 뒤섞이고, 리팩터링이나 확장 중에 기존 동작이 조용히 깨질 수 있다.

계약을 먼저 정하면 구현은 그 계약을 만족하는 수단이 된다. 학습자는 클래스나 알고리즘을 먼저 떠올리기보다, 유효한 입력, 고정 비율, 실패 조건, 출력 보존, 확장 불변성을 먼저 생각하게 된다. 이것이 테스트 주도와 클린 아키텍처 학습의 출발점이다.

## 4. 문제 정의

### 표면 문제

meter, feet, yard 사이의 길이 값을 변환하고 결과를 출력해야 한다. 잘못된 입력이나 지원하지 않는 단위는 변환하지 않아야 한다.

### 개선된 정의

이 문제는 길이 단위 변환이라는 작은 도메인을 사용해, 변환 비율과 실패 조건을 테스트 가능한 계약으로 고정하는 학습 문제다. 시스템은 meter, feet, yard의 기준 비율을 안정적으로 유지해야 하며, 입력 형식과 숫자 값, 지원 단위 여부를 명확히 검증해야 한다. 출력은 변환 결과뿐 아니라 원 입력 값과 원 입력 단위를 보존해야 한다. 설정 외부화나 동적 단위 등록이 추가되더라도 기존 기본 단위의 비율 계약은 변경되지 않아야 한다. JSON, CSV, 표 형태 같은 출력 포맷은 계산 결과를 바꾸지 않고 표현만 달라져야 한다. 따라서 핵심은 변환 공식 구현이 아니라, 확장 가능한 요구사항을 책임 분리와 회귀 테스트 관점으로 정의하는 것이다.

### 핵심 Invariant

- meter, feet, yard의 기본 비율은 명시된 기준 없이 변경되지 않는다.
- feet와 yard 사이의 변환은 meter 기준 비율을 통해 일관되게 계산된다.
- 유효하지 않은 입력은 변환 결과를 생성하지 않는다.
- 출력은 원 입력 값과 원 입력 단위를 보존한다.
- 새 단위나 새 출력 포맷이 추가되어도 기존 기본 단위 변환 결과는 변하지 않는다.

### 훈련하려는 사고 능력

- 요구사항을 테스트 가능한 계약으로 바꾸는 능력
- 계산 책임, 입력 검증 책임, 출력 표현 책임을 분리하는 능력
- 확장 요구가 기존 동작을 깨뜨리지 않도록 불변 조건을 세우는 능력

## 5. Mom Test 질문

1. 최근에 단순한 계산 문제라고 생각하고 시작했다가, 입력 검증이나 예외 처리 때문에 시간이 늘어난 경험이 있었나요? 그때 실제로 어떤 순서로 작업했고, 어디서 시간을 가장 많이 썼나요? 그 문제가 과제 제출, 코드 리뷰, 리팩터링 실패 같은 리스크로 이어졌나요?

2. 마지막으로 단위 변환, 포맷 변환, 계산기 같은 작은 프로그램을 만들었을 때 테스트를 먼저 작성했나요, 나중에 작성했나요? 실제로 어떤 케이스를 테스트했고 어떤 케이스는 놓쳤나요? 놓친 케이스 때문에 재작업 시간이 얼마나 들었나요?

3. 최근에 “이건 그냥 if문 몇 개면 되겠다”고 판단했다가 요구사항이 늘어난 경험이 있나요? 그때 실제로 코드를 어떻게 바꿨고, 기존 동작이 깨졌는지 확인하는 데 얼마나 걸렸나요? 그 변경이 성적, 리뷰 피드백, 일정 지연에 영향을 줬나요?

4. 과거에 입력 형식이 애매해서 구현 중간에 기준을 다시 정한 적이 있나요? 그때 유효한 입력과 잘못된 입력을 어떻게 구분했나요? 기준이 늦게 정해져서 테스트나 문서를 다시 고친 시간이 있었나요?

5. 최근 프로젝트에서 README나 요구사항 문서에 적힌 숫자, 규칙, 포맷을 코드와 맞추지 못한 적이 있었나요? 실제로 어떤 불일치가 발생했고 어떻게 발견했나요? 그 불일치가 버그, 감점, 리뷰 지적, 신뢰 저하로 이어졌나요?

6. 마지막으로 출력 형식을 바꿔야 했던 경험이 있나요? 예를 들어 콘솔 출력, JSON, CSV처럼 표현 방식이 달라졌을 때 계산 로직까지 같이 수정했나요? 그 수정 때문에 기존 결과를 다시 검증하는 데 시간이 얼마나 들었나요?

7. 최근에 새 옵션, 새 타입, 새 단위처럼 “하나만 더 추가”하는 변경을 해본 적이 있나요? 실제로 추가할 때 기존 코드를 몇 군데나 고쳤나요? 그 변경 범위가 커져서 실수하거나 테스트를 늘려야 했던 비용이 있었나요?

8. 과거에 설정 파일이나 외부 입력 값이 잘못되어 프로그램이 조용히 틀린 결과를 낸 경험이 있나요? 그때 실제로 오류를 어떻게 감지했고, 사용자는 어떤 행동을 했나요? 잘못된 결과를 뒤늦게 고치는 데 시간이나 신뢰 비용이 들었나요?

9. 최근에 클린 아키텍처, OCP, SRP 같은 원칙을 배웠을 때 실제 과제 코드에 적용해 본 경험이 있나요? 적용했다면 어떤 책임을 나눴고, 적용하지 못했다면 어디서 막혔나요? 그 차이가 코드 리뷰, 리팩터링 시간, 테스트 작성 시간에 영향을 줬나요?

10. 마지막으로 테스트가 없어서 리팩터링을 망설였던 경험이 있나요? 그때 실제로 어떤 변경을 미뤘거나 작게 쪼갰나요? 테스트 부재가 학습 속도, 제출 일정, 버그 수정 리스크에 어떤 영향을 줬나요?

## 6. 진짜 위험 가정

- 학습자는 단위 변환 계산 자체보다 입력 검증, 계약 테스트, 책임 분리 학습에 실제 시간을 쓰고 싶어 한다.
- 학습자는 OCP/SRP 같은 설계 원칙을 추상 설명이 아니라 작은 실습 문제 안에서 경험해야 더 잘 이해한다.
- 학습자는 테스트를 “추가 작업”이 아니라 리팩터링과 확장을 안전하게 만드는 시간 절감 장치로 받아들일 준비가 되어 있다.
