# 23_커버리지 점검 프롬프트 보고서

## 1. 작업 개요

이번 세션에서는 `UnitConverter_A02` 프로젝트의 현재 테스트 커버리지를 점검했다.

사용자가 요청한 확인 범위는 다음과 같았다.

- JaCoCo 리포트 생성: `mvn jacoco:report`
- 생성 리포트 확인: `target/site/jacoco/index.html`
- 커버되지 않은 줄 중 Invariant 관련 분기 우선 식별
- 특히 `meter`↔`feet`/`yard` 비율 변환 분기, 음수 입력 처리, 없는 단위 처리 확인
- Domain Logic 95% 미달 시 누락 테스트 케이스 제안
- Boundary 85% 미달 시 누락 계약 테스트 케이스 제안
- 결과를 `| 파일 | Stmts | Miss | Cover | 미달 원인 | 추가 테스트 제안 |` 형식으로 정리

이번 세션에서는 구현 코드나 테스트 코드를 추가하지 않고, 현재 리포지토리의 커버리지 산출물과 테스트 파일을 기준으로 분석했다.

## 6. Invariant 관련 분기 확인

요청된 Invariant 관련 항목은 `UnitConverterService.java` 기준으로 다음과 같이 확인했다.

### 6.1 `meter`↔`feet`/`yard` 비율 변환

`UnitConverterServiceTest.java`에 다음 계약 테스트가 존재했다.

- `test_normalConversion_meterToFeet_returnsExpectedFeet`
- `test_normalConversion_meterToYard_returnsExpectedYard`
- `test_normalConversion_feetToMeter_returnsReverseMeter`
- `test_normalConversion_yardToMeter_returnsReverseMeter`
- `test_normalConversion_convertAllMeter_returnsRegisteredTargetConversions`

따라서 활성 서비스 클래스 기준으로 `meter`↔`feet`/`yard` 변환 라인은 커버되어 있다.

다만 `src/main/java/UnitConverter.java`는 별도 레거시 클래스처럼 남아 있으며, 이 클래스의 `meter -> feet` 분기와 unsupported 분기는 전혀 커버되지 않았다.

### 6.2 음수 입력 처리

`UnitConverterService.java`의 `validateNonNegative`는 라인 및 브랜치 기준 모두 커버되었다.

관련 테스트는 다음과 같다.

- `test_exception_negativeValue_throwsIllegalArgumentException`

따라서 음수 입력 Invariant는 서비스 경로 기준으로 커버되어 있다.

### 6.3 없는 단위 처리

`UnitConverterService.java`의 `ratioFor`에서 없는 단위에 대한 예외 분기는 커버되었다.

관련 테스트는 다음과 같다.

- `test_exception_unknownUnit_throwsIllegalArgumentException`

따라서 없는 단위 처리 Invariant도 서비스 경로 기준으로 커버되어 있다.

## 7. 부분 분기 미커버

`UnitConverterService.java`에서 라인은 모두 커버되었지만 다음 조건식에는 부분 분기 미커버가 남아 있었다.

- `loadConfig`: `.yaml` 또는 `.yml` 확장자 판별 중 `.yml` 경로
- `parseInput`: `input == null || colon count != 1` 중 `null` 입력 경로
- `parseJson`: `unit`/`metersPerUnit` 한쪽만 존재하는 객체 조합 일부
- `parseYaml`: `currentUnit == null || currentUnit.isBlank()` 중 빈 unit 경로
- `validateUnitName`: `unit == null || unit.isBlank()` 중 `null` 단위 경로

이 항목들은 라인 커버리지 목표 미달 원인은 아니지만, 경계값 및 실패 계약을 더 명확히 만들 수 있는 후보로 판단했다.

## 8. Domain / Boundary 기준 판정

현재 코드 구조에서는 Domain과 Boundary가 패키지나 클래스 단위로 명확히 분리되어 있지 않다.

따라서 Domain Logic과 Boundary 커버리지를 독립 산정하기는 어렵다.

다만 활성 서비스 표면인 `UnitConverterService.java` 기준으로 보면 라인 커버리지는 100%이므로, 사용자가 지정한 다음 기준은 라인 기준으로 미달하지 않는다.

- Domain Logic 95% 이상: 미달 아님
- Boundary 85% 이상: 미달 아님

따라서 "미달 시" 요구된 필수 누락 테스트 케이스 제안은 트리거되지 않았다.

다만 브랜치 커버리지 관점에서는 전체 84.4%로 85%에 근접하며, `UnitConverterService.java`의 부분 분기와 `UnitConverter.java` 0% 커버리지가 주요 리스크다.
